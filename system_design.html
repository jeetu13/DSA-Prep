S Single Responsibility Principle

▼

Definition: "A class should have only one reason to change." — Robert C. Martin

In practice: every class, function, or module should focus on a single concern. This makes code easier to understand, test, and modify.

❌ Violation: A User class that handles authentication, profile management, email sending, and database persistence — it has many reasons to change.
❌ BAD — violates SRP
// This class does TOO many things
public class User {
    private String name;
    private String email;

    // Reason 1: User data validation
    public boolean validateEmail() {
        return email.contains("@");
    }

    // Reason 2: Database persistence
    public void saveToDatabase() {
        // SQL logic here
        System.out.println("INSERT INTO users...");
    }

    // Reason 3: Email notification
    public void sendWelcomeEmail() {
        // SMTP logic here
        System.out.println("Sending email to: " + email);
    }

    // Reason 4: Report generation
    public String generateReport() {
        return "User Report: " + name;
    }
}
✅ GOOD — follows SRP
// Each class has one clear responsibility

public class User {                          // Holds user data only
    private String name;
    private String email;
    public String getName()  { return name;  }
    public String getEmail() { return email; }
}

public class UserValidator {               // Validation logic
    public boolean isValidEmail(User user) {
        return user.getEmail().contains("@");
    }
}

public class UserRepository {             // DB persistence
    public void save(User user) {
        // SQL INSERT logic
    }
    public User findById(int id) { return null; }
}

public class EmailService {               // Email sending
    public void sendWelcome(User user) {
        // SMTP logic
    }
}

public class UserReportGenerator {         // Report generation
    public String generate(User user) {
        return "User Report: " + user.getName();
    }
}
✅ Industry Context: In microservices, SRP maps directly to service boundaries. Each service (UserService, NotificationService, ReportingService) has a single business capability. Spring Boot services naturally enforce SRP through @Service, @Repository, @Component separation.

O Open/Closed Principle

▼

Definition: Software entities should be open for extension but closed for modification.

Achieve this using abstractions (interfaces/abstract classes) so new behavior can be added by creating new classes rather than modifying existing ones.

❌ Violation: Every time you add a new payment method to a PaymentProcessor, you modify the existing switch/if chain — breaking OCP.
// ❌ BAD: Must modify this class every time new payment type is added
public class PaymentProcessor {
    public void processPayment(String type, double amount) {
        if (type.equals("CREDIT_CARD")) {
            System.out.println("Processing credit card: $" + amount);
        } else if (type.equals("PAYPAL")) {
            System.out.println("Processing PayPal: $" + amount);
        } else if (type.equals("CRYPTO")) {   // Added later — forces modification!
            System.out.println("Processing Crypto: $" + amount);
        }
    }
}
// ✅ GOOD: Open for extension (new class), closed for modification

public interface PaymentMethod {
    void process(double amount);
}

public class CreditCardPayment implements PaymentMethod {
    @Override
    public void process(double amount) {
        System.out.println("Charging credit card: $" + amount);
        // Stripe/Braintree integration
    }
}

public class PayPalPayment implements PaymentMethod {
    @Override
    public void process(double amount) {
        System.out.println("PayPal transfer: $" + amount);
    }
}

// Adding Crypto? Just create a new class — no existing code changes!
public class CryptoPayment implements PaymentMethod {
    @Override
    public void process(double amount) {
        System.out.println("Bitcoin transfer: $" + amount);
    }
}

public class PaymentProcessor {
    public void processPayment(PaymentMethod method, double amount) {
        method.process(amount);   // Never needs to change
    }
}
✅ Real World: Java's Comparator interface, Spring's HandlerInterceptor, and JPA's Specification pattern all follow OCP — you extend behavior without touching framework code.

L Liskov Substitution Principle

▼

Definition: If S is a subtype of T, then objects of type T may be replaced with objects of type S without altering the correctness of the program.

In simple terms: derived classes must be fully substitutable for base classes. A subclass should honor the contract of the parent class.

❌ Classic Violation: Square extends Rectangle. Setting width on a Square secretly changes height too — breaks the Rectangle contract.
// ❌ The classic Rectangle–Square violation
public class Rectangle {
    protected int width, height;
    public void setWidth(int w)  { this.width  = w; }
    public void setHeight(int h) { this.height = h; }
    public int  area()            { return width * height; }
}

public class Square extends Rectangle {
    @Override
    public void setWidth(int w)  { this.width = this.height = w; } // 💥 Surprise!
    @Override
    public void setHeight(int h) { this.width = this.height = h; } // 💥 Surprise!
}

// Code that works for Rectangle breaks for Square!
void resize(Rectangle r) {
    r.setWidth(5);
    r.setHeight(10);
    // Expected area = 50, but Square gives 100 — LSP violated!
    assert r.area() == 50;
}
// ✅ GOOD: Model shapes independently using abstraction
public interface Shape {
    int area();
}

public class Rectangle implements Shape {
    private int width, height;
    public Rectangle(int w, int h) { width = w; height = h; }
    @Override public int area() { return width * height; }
}

public class Square implements Shape {
    private int side;
    public Square(int s) { side = s; }
    @Override public int area() { return side * side; }
}
// Now both are interchangeable via Shape — no surprises!
✅ Rule of thumb: If you find yourself using instanceof checks in methods that accept a base type, you're likely violating LSP. Fix by refactoring the hierarchy or using polymorphism properly.

I Interface Segregation Principle

▼

Definition: Clients should not be forced to depend on interfaces they do not use. Prefer multiple specific interfaces over one general-purpose "fat" interface.

❌ Violation: A single Worker interface with methods work(), eat(), sleep() forces Robot classes to implement eat() and sleep() with empty/error bodies.
// ❌ BAD: Fat interface forces irrelevant methods
public interface Worker {
    void work();
    void eat();    // Robots don't eat!
    void sleep();  // Robots don't sleep!
}

public class Robot implements Worker {
    @Override public void work()  { System.out.println("Working 24/7"); }
    @Override public void eat()   { throw new UnsupportedOperationException(); } // 💥
    @Override public void sleep() { throw new UnsupportedOperationException(); } // 💥
}
// ✅ GOOD: Segregated interfaces
public interface Workable  { void work();  }
public interface Eatable   { void eat();   }
public interface Sleepable { void sleep(); }

public class HumanWorker implements Workable, Eatable, Sleepable {
    @Override public void work()  { System.out.println("Human working");  }
    @Override public void eat()   { System.out.println("Human eating");   }
    @Override public void sleep() { System.out.println("Human sleeping"); }
}

public class Robot implements Workable {  // Only what it needs!
    @Override public void work() { System.out.println("Robot working 24/7"); }
}

// Industry example: Java's Serializable, Comparable, Cloneable are all segregated
// Spring's ApplicationListener<E>, InitializingBean, DisposableBean are ISP-compliant
💡 Test: If implementing an interface forces you to write throw new UnsupportedOperationException() — ISP is being violated. Break the interface up.

D Dependency Inversion Principle

▼

Definition: High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details; details should depend on abstractions.

// ❌ BAD: OrderService directly depends on concrete MySQLDatabase
public class MySQLDatabase {
    public void save(String data) { System.out.println("MySQL: " + data); }
}

public class OrderService {
    private MySQLDatabase db = new MySQLDatabase();  // Tightly coupled! 💥
    public void placeOrder(String order) {
        db.save(order);
    }
}
// ✅ GOOD: Both depend on the Database abstraction
public interface Database {
    void save(String data);
}

public class MySQLDatabase implements Database {
    @Override
    public void save(String data) { System.out.println("MySQL: " + data); }
}

public class MongoDatabase implements Database {
    @Override
    public void save(String data) { System.out.println("MongoDB: " + data); }
}

public class OrderService {
    private final Database db;    // Depends on abstraction!

    public OrderService(Database db) {  // Injected via constructor (Spring @Autowired)
        this.db = db;
    }

    public void placeOrder(String order) {
        db.save(order);   // Switch from MySQL to Mongo without touching this class!
    }
}

// Usage — Spring Boot does this automatically with @Autowired
Database db = new MySQLDatabase();
OrderService service = new OrderService(db);
✅ Spring Context: Spring's IoC container IS dependency inversion at scale. @Service, @Repository, @Component — you wire abstractions, Spring provides implementations at runtime. This is DIP applied to an entire framework.

Singleton Ensure only one instance exists globally

▼

Intent: Ensure a class has only one instance, and provide a global access point to it.

Use when: Configuration managers, Logger, Thread pool, Database connection pool, Cache manager.

Thread-Safe Singleton (Double-Checked Locking)
public class ConfigurationManager {

    // volatile ensures visibility across threads (Java Memory Model)
    private static volatile ConfigurationManager instance;
    private Properties config;

    private ConfigurationManager() {    // Private constructor — no external new
        config = new Properties();
        config.setProperty("db.host", "localhost");
        config.setProperty("db.port", "5432");
    }

    // Double-checked locking for thread safety with performance
    public static ConfigurationManager getInstance() {
        if (instance == null) {                    // First check (no sync)
            synchronized (ConfigurationManager.class) {
                if (instance == null) {             // Second check (with sync)
                    instance = new ConfigurationManager();
                }
            }
        }
        return instance;
    }

    public String get(String key) { return config.getProperty(key); }
}

// Alternatively — the Enum Singleton (Bill Pugh / most preferred)
public enum AppConfig {
    INSTANCE;
    private String dbHost = "localhost";
    public String getDbHost() { return dbHost; }
}
// Usage: AppConfig.INSTANCE.getDbHost()
// Enum singletons are: serialization-safe + reflection-safe + thread-safe
⚠️ Pitfalls: Singletons make unit testing hard (global state). In Spring, use @Scope("singleton") beans instead — Spring manages the lifecycle and allows mocking in tests.
✅ Spring equiv: All Spring beans are singletons by default. @Component, @Service, @Repository — one instance per Spring context.

Factory Method Delegate object creation to subclasses

▼

Intent: Define an interface for creating an object, but let subclasses decide which class to instantiate. Factory Method lets a class defer instantiation to subclasses.

// Real-world: Payment gateway factory for a fintech app

public interface PaymentGateway {
    void processPayment(double amount);
    boolean refund(String transactionId);
}

public class StripeGateway implements PaymentGateway {
    @Override
    public void processPayment(double amount) {
        System.out.println("Stripe charging: $" + amount);
    }
    @Override
    public boolean refund(String txId) { return true; }
}

public class RazorpayGateway implements PaymentGateway {
    @Override
    public void processPayment(double amount) {
        System.out.println("Razorpay charging: ₹" + amount);
    }
    @Override
    public boolean refund(String txId) { return true; }
}

// Factory — centralizes creation logic
public class PaymentGatewayFactory {
    public static PaymentGateway create(String region) {
        return switch (region.toUpperCase()) {
            case "US", "EU" -> new StripeGateway();
            case "IN"        -> new RazorpayGateway();
            default -> throw new IllegalArgumentException("Unknown region: " + region);
        };
    }
}

// Usage — client code is decoupled from concrete classes
PaymentGateway gateway = PaymentGatewayFactory.create("IN");
gateway.processPayment(1500.0);
💡 Real usage: java.util.Calendar.getInstance(), Connection = DriverManager.getConnection(), Spring's BeanFactory, LoggerFactory.getLogger() in SLF4J.

Builder Construct complex objects step by step

▼

Intent: Separate the construction of a complex object from its representation. Use when an object has many optional parameters (avoids telescoping constructors).

// Real-world: HTTP Request builder for a telecom API client

public class HttpRequest {
    private final String              url;
    private final String              method;
    private final Map<String,String>   headers;
    private final String              body;
    private final int                 timeoutMs;
    private final boolean             followRedirects;

    private HttpRequest(Builder b) {  // Private — only Builder can create
        this.url             = b.url;
        this.method          = b.method;
        this.headers         = Collections.unmodifiableMap(b.headers);
        this.body            = b.body;
        this.timeoutMs       = b.timeoutMs;
        this.followRedirects = b.followRedirects;
    }

    public static class Builder {
        private final String             url;      // Required
        private       String             method          = "GET";
        private       Map<String,String> headers         = new HashMap<>();
        private       String             body            = null;
        private       int               timeoutMs       = 5000;
        private       boolean           followRedirects = true;

        public Builder(String url) { this.url = url; }

        public Builder method(String m)           { method = m; return this; }
        public Builder header(String k, String v) { headers.put(k,v); return this; }
        public Builder body(String b)             { body = b; return this; }
        public Builder timeout(int ms)            { timeoutMs = ms; return this; }
        public Builder noRedirects()              { followRedirects = false; return this; }

        public HttpRequest build() {
            if (url == null || url.isBlank()) throw new IllegalStateException("URL required");
            return new HttpRequest(this);
        }
    }
}

// Beautiful fluent API — clear, readable, no constructor telescope
HttpRequest req = new HttpRequest.Builder("https://api.mobileum.com/cdr")
    .method("POST")
    .header("Authorization", "Bearer token123")
    .header("Content-Type", "application/json")
    .body("{\"msisdn\":\"9876543210\"}")
    .timeout(3000)
    .build();
✅ Used everywhere: StringBuilder, Stream.Builder, Lombok's @Builder, Spring's UriComponentsBuilder, MockMvcRequestBuilders, Hibernate Criteria Builder.

Prototype Clone existing objects instead of new creation

▼

Intent: Create new objects by copying (cloning) an existing object. Useful when object creation is expensive (DB fetch, complex init).

public class TelecomSubscriber implements Cloneable {
    private String       msisdn;
    private String       plan;
    private List<String> services;  // e.g. ["VoLTE","VoWifi","RCS"]

    public TelecomSubscriber(String msisdn, String plan, List<String> services) {
        this.msisdn   = msisdn;
        this.plan     = plan;
        this.services = new ArrayList<>(services);  // defensive copy
    }

    // Deep clone — own copy of mutable fields
    @Override
    public TelecomSubscriber clone() {
        try {
            TelecomSubscriber copy = (TelecomSubscriber) super.clone();
            copy.services = new ArrayList<>(this.services);  // deep copy list
            return copy;
        } catch (CloneNotSupportedException e) {
            throw new RuntimeException(e);
        }
    }
}

// Clone a template subscriber for batch provisioning — avoids DB hit each time
TelecomSubscriber template = new TelecomSubscriber("BASE", "5G-PREMIUM",
                                  List.of("VoLTE", "VoWifi"));

TelecomSubscriber sub1 = template.clone(); sub1.msisdn = "9876543210";
TelecomSubscriber sub2 = template.clone(); sub2.msisdn = "9876543211";
💡 Real usage: Spring's BeanDefinition prototype scope, Object.clone(), copy constructors, record copy-with patterns in Java 16+.

Abstract Factory Create families of related objects

▼

Intent: Provide an interface for creating families of related or dependent objects without specifying their concrete classes. Think of it as a "factory of factories."

// Real-world: Multi-cloud infrastructure provisioning

public interface VirtualMachine { void start(); }
public interface Database       { void connect(); }
public interface Storage        { void upload(String file); }

// Abstract Factory — creates a "family" of cloud resources
public interface CloudProvider {
    VirtualMachine createVM();
    Database       createDatabase();
    Storage        createStorage();
}

// AWS family
public class AWSEC2      implements VirtualMachine { @Override public void start()           { System.out.println("EC2 starting");   }}
public class AWSRDS      implements Database       { @Override public void connect()         { System.out.println("RDS connected");   }}
public class AWSS3       implements Storage        { @Override public void upload(String f) { System.out.println("S3 upload: "+f); }}

public class AWSProvider implements CloudProvider {
    @Override public VirtualMachine createVM()            { return new AWSEC2();  }
    @Override public Database       createDatabase()       { return new AWSRDS();  }
    @Override public Storage        createStorage()        { return new AWSS3();   }
}

// GCP family (similarly defined)
public class GCPProvider implements CloudProvider { /* GCE, CloudSQL, GCS */ }

// Application code — zero knowledge of AWS vs GCP
public class InfraProvisioner {
    private final CloudProvider cloud;
    public InfraProvisioner(CloudProvider cloud) { this.cloud = cloud; }

    public void provision() {
        cloud.createVM().start();
        cloud.createDatabase().connect();
        cloud.createStorage().upload("config.yaml");
    }
}

new InfraProvisioner(new AWSProvider()).provision();

Adapter Make incompatible interfaces work together

▼

Intent: Convert the interface of a class into another interface clients expect. Lets classes with incompatible interfaces collaborate.

Real-world analogy: A power adapter that lets you plug a US device into a European socket.

// Real-world: Integrating a legacy SOAP-based telecom BSS with a modern REST service

// New interface our app expects
public interface SubscriberService {
    String getSubscriberInfo(String msisdn);
}

// Legacy SOAP client — can't change this (external library)
public class LegacyBSSClient {
    public String fetchSubscriberBySoap(String imsi, String authToken) {
        return "<subscriber><imsi>" + imsi + "</imsi></subscriber>";
    }
}

// Adapter — wraps legacy, implements modern interface
public class LegacyBSSAdapter implements SubscriberService {
    private final LegacyBSSClient legacyClient;
    private final String          authToken;

    public LegacyBSSAdapter(LegacyBSSClient client, String token) {
        this.legacyClient = client;
        this.authToken    = token;
    }

    @Override
    public String getSubscriberInfo(String msisdn) {
        String imsi = convertMsisdnToImsi(msisdn);      // Format translation
        String xml  = legacyClient.fetchSubscriberBySoap(imsi, authToken);
        return parseXmlToJson(xml);                   // Protocol translation
    }

    private String convertMsisdnToImsi(String msisdn) { return "404-" + msisdn; }
    private String parseXmlToJson(String xml) { return "{\"msisdn\":\"mapped\"}"; }
}

// Client works with modern interface — agnostic of legacy SOAP
SubscriberService service = new LegacyBSSAdapter(new LegacyBSSClient(), "token");
String info = service.getSubscriberInfo("9876543210");
✅ Real usage: Arrays.asList() adapts array to List, InputStreamReader adapts InputStream to Reader, Spring's HandlerAdapter, JDBC adapters for different databases.

Decorator Add behavior to objects dynamically

▼

Intent: Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.

// Real-world: Telecom notification system with logging, retry, encryption

public interface NotificationSender {
    void send(String recipient, String message);
}

// Base component
public class SMSSender implements NotificationSender {
    @Override
    public void send(String to, String msg) {
        System.out.printf("SMS → %s: %s%n", to, msg);
    }
}

// Base decorator
public abstract class NotificationDecorator implements NotificationSender {
    protected final NotificationSender wrapped;
    protected NotificationDecorator(NotificationSender n) { wrapped = n; }
}

// Logging decorator
public class LoggingDecorator extends NotificationDecorator {
    public LoggingDecorator(NotificationSender n) { super(n); }
    @Override
    public void send(String to, String msg) {
        System.out.println("[LOG] Sending to: " + to);
        wrapped.send(to, msg);
        System.out.println("[LOG] Sent successfully");
    }
}

// Retry decorator
public class RetryDecorator extends NotificationDecorator {
    private final int maxRetries;
    public RetryDecorator(NotificationSender n, int retries) {
        super(n); maxRetries = retries;
    }
    @Override
    public void send(String to, String msg) {
        for (int i = 0; i < maxRetries; i++) {
            try { wrapped.send(to, msg); return; }
            catch (Exception e) { System.out.println("Retry "+(i+1)); }
        }
    }
}

// Compose decorators — order matters!
NotificationSender sender =
    new LoggingDecorator(
        new RetryDecorator(
            new SMSSender(), 3   // Base wrapped with Retry wrapped with Logging
        )
    );
sender.send("9876543210", "Your OTP is 456789");
✅ Used in Java I/O: new BufferedReader(new InputStreamReader(new FileInputStream("file"))) — that IS the Decorator pattern. Spring Security filter chain is Decorator-based.

Facade Simplified interface to a complex subsystem

▼

Intent: Provide a unified, simplified interface to a set of interfaces in a subsystem. Makes the subsystem easier to use.

// Real-world: Telecom subscriber onboarding — orchestrates multiple subsystems

public class BSSService      { public void createAccount(String msisdn)  { /* BSS */  }}
public class HSSService      { public void provisionSIM(String msisdn)   { /* HSS */  }}
public class PCRFService     { public void applyPolicy(String msisdn)    { /* PCRF */ }}
public class BillingService  { public void createWallet(String msisdn)  { /* OCS */  }}
public class SMSService      { public void sendWelcome(String msisdn)    { /* SMS */  }}

// FACADE — single entry point for the entire onboarding workflow
public class SubscriberOnboardingFacade {
    private final BSSService     bss;
    private final HSSService     hss;
    private final PCRFService    pcrf;
    private final BillingService billing;
    private final SMSService     sms;

    public SubscriberOnboardingFacade() {
        bss     = new BSSService();
        hss     = new HSSService();
        pcrf    = new PCRFService();
        billing = new BillingService();
        sms     = new SMSService();
    }

    // One method — client doesn't know about 5 subsystems
    public void onboardSubscriber(String msisdn) {
        System.out.println("Onboarding: " + msisdn);
        bss.createAccount(msisdn);
        hss.provisionSIM(msisdn);
        pcrf.applyPolicy(msisdn);
        billing.createWallet(msisdn);
        sms.sendWelcome(msisdn);
        System.out.println("Onboarding complete!");
    }
}

// Client: one simple call orchestrates everything
new SubscriberOnboardingFacade().onboardSubscriber("9876543210");
💡 Real usage: Spring's JdbcTemplate is a Facade over JDBC complexity. SLF4J is a Facade over logging frameworks. Spring Boot's @SpringBootApplication is a Facade over component scan + auto-configuration + property loading.

Proxy Surrogate / placeholder to control access

▼

Intent: Provide a surrogate or placeholder for another object to control access to it. Types: Virtual (lazy loading), Protection (access control), Remote (network proxy), Caching.

// Caching Proxy — avoid repeated expensive DB calls
public interface CDRRepository {
    List<String> getCDRs(String msisdn);
}

public class DatabaseCDRRepository implements CDRRepository {
    @Override
    public List<String> getCDRs(String msisdn) {
        System.out.println("[DB] Heavy query for: " + msisdn);
        return List.of("CDR_001", "CDR_002", "CDR_003");  // Expensive!
    }
}

// Caching Proxy — transparent to the caller
public class CachingCDRProxy implements CDRRepository {
    private final CDRRepository              real;
    private final Map<String, List<String>>  cache = new HashMap<>();

    public CachingCDRProxy(CDRRepository real) { this.real = real; }

    @Override
    public List<String> getCDRs(String msisdn) {
        if (!cache.containsKey(msisdn)) {
            System.out.println("[CACHE MISS] Fetching from DB");
            cache.put(msisdn, real.getCDRs(msisdn));
        } else {
            System.out.println("[CACHE HIT]  Returning cached");
        }
        return cache.get(msisdn);
    }
}

// Spring AOP creates dynamic proxies for @Transactional, @Cacheable, @Secured
CDRRepository repo = new CachingCDRProxy(new DatabaseCDRRepository());
✅ Spring is Proxy-heavy: @Transactional wraps beans in a JDK dynamic proxy/CGLIB proxy. @Cacheable, @Async, @Secured — all implemented via proxies. Hibernate's lazy loading uses proxies for entities.

Composite Tree structures — treat individual and group uniformly

▼

Intent: Compose objects into tree structures to represent part-whole hierarchies. Lets clients treat individual objects and compositions of objects uniformly.

// Real-world: Telecom network hierarchy — Node (BTS/BSC/MSC) tree
public interface NetworkNode {
    void   display(String indent);
    int    getCapacity();
}

// Leaf — BTS (Base Transceiver Station)
public class BTS implements NetworkNode {
    private String id;
    private int    capacity;
    public BTS(String id, int cap) { this.id = id; this.capacity = cap; }
    @Override public void display(String indent) { System.out.println(indent + "BTS[" + id + "] cap=" + capacity); }
    @Override public int  getCapacity() { return capacity; }
}

// Composite — BSC manages multiple BTSes
public class BSC implements NetworkNode {
    private String              id;
    private List<NetworkNode>  children = new ArrayList<>();

Strategy Define a family of algorithms, make them interchangeable

▼

Intent: Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from clients that use it.

// Real-world: Telecom fraud detection — different detection strategies

public interface FraudDetectionStrategy {
    boolean isFraudulent(CallRecord cdr);
}

public class HighVolumeStrategy implements FraudDetectionStrategy {
    private final int threshold;
    public HighVolumeStrategy(int t) { threshold = t; }
    @Override
    public boolean isFraudulent(CallRecord cdr) {
        return cdr.getCallCount() > threshold;   // >500 calls/hour = fraud
    }
}

public class GeoAnomalyStrategy implements FraudDetectionStrategy {
    @Override
    public boolean isFraudulent(CallRecord cdr) {
        return cdr.getOriginCountry().equals("HIGH_RISK");
    }
}

public class MLModelStrategy implements FraudDetectionStrategy {
    @Override
    public boolean isFraudulent(CallRecord cdr) {
        return callMLModel(cdr) > 0.85;    // >85% fraud probability

Chain of Responsibility Pass requests along a chain of handlers

▼

Intent: Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle it. Chain the receiving objects and pass the request along until one handles it.

// Real-world: HTTP middleware pipeline (like Spring Security filter chain)

public abstract class RequestHandler {
    protected RequestHandler next;

    public RequestHandler setNext(RequestHandler handler) {
        this.next = handler;
        return handler;    // enables chaining: a.setNext(b).setNext(c)
    }

    public abstract void handle(HttpRequest request);
}

public class AuthenticationHandler extends RequestHandler {
    @Override
    public void handle(HttpRequest request) {
        if (request.getHeader("Authorization") == null) {
            System.out.println("[AUTH] REJECTED: No token"); return;
        }
        System.out.println("[AUTH] Passed");
        if (next != null) next.handle(request);
    }
}

public class RateLimitHandler extends RequestHandler {
    @Override
    public void handle(HttpRequest request) {
        System.out.println("[RATE] Checking rate limit...");
        // Check Redis for rate limit bucket
        if (next != null) next.handle(request);
    }
}

public class LoggingHandler extends RequestHandler {
    @Override
    public void handle(HttpRequest request) {
        System.out.println("[LOG] Request: " + request.getUrl());
        if (next != null) next.handle(request);
    }
}

// Build the chain
RequestHandler auth = new AuthenticationHandler();
auth.setNext(new RateLimitHandler())
    .setNext(new LoggingHandler());

auth.handle(request);  // Auth → RateLimit → Logging → Business Logic
💡 This IS Spring Security: SecurityFilterChain is Chain of Responsibility. Each filter (JwtFilter, CorsFilter, CsrfFilter) processes the request and passes it to the next.

Template Method Define algorithm skeleton, defer steps to subclasses

▼

Intent: Define the skeleton of an algorithm in the base class, letting subclasses redefine specific steps without changing the algorithm's structure.

// Real-world: CDR (Call Detail Record) processing pipeline

public abstract class CDRProcessor {

    // Template method — defines the invariant workflow
    public final void process(String rawCDR) {
        String parsed    = parseCDR(rawCDR);        // Step 1: Parse
        String validated = validate(parsed);        // Step 2: Validate
        String enriched  = enrich(validated);       // Step 3: Enrich (hook)
        store(enriched);                            // Step 4: Store
    }

    protected abstract String parseCDR(String raw);
    protected abstract String validate(String cdr);

    // Hook — optional override (has default impl)
    protected String enrich(String cdr) { return cdr; }
    protected abstract void store(String cdr);
}

public class VoiceCDRProcessor extends CDRProcessor {
    @Override protected String parseCDR(String raw)    { return "VOICE:" + raw;  }
    @Override protected String validate(String cdr)   { return cdr;             }
    @Override protected String enrich(String cdr)     { return cdr + "+GEO";   }   // adds geo data
    @Override protected void   store(String cdr)      { System.out.println("Voice DB: " + cdr); }
}

public class DataCDRProcessor extends CDRProcessor {
    @Override protected String parseCDR(String raw)    { return "DATA:" + raw;   }
    @Override protected String validate(String cdr)   { return cdr;             }
    @Override protected void   store(String cdr)      { System.out.println("Data Lake: " + cdr); }
}

Command Encapsulate requests as objects (undo/queue/log)

Intent: Encapsulate a request as an object, letting you parameterize clients with different requests, queue or log requests, and support undoable operations.

public interface Command {
    void execute();
    void undo();
}

public class ActivateServiceCommand implements Command {
    private final String msisdn;
    private final String service;

    public ActivateServiceCommand(String msisdn, String service) {
        this.msisdn  = msisdn;
        this.service = service;
    }
    @Override
    public void execute() { System.out.println("✅ Activating " + service + " for " + msisdn); }
    @Override
    public void undo()    { System.out.println("↩️  Deactivating " + service + " for " + msisdn); }
}

// Invoker — with undo history
public class ServiceManager {
    private final Deque<Command> history = new ArrayDeque<>();

    public void execute(Command cmd) {
        cmd.execute();
        history.push(cmd);
    }

    public void undoLast() {
        if (!history.isEmpty()) history.pop().undo();
    }
}

mgr.execute(new ActivateServiceCommand("9876543210", "VoWiFi"));

Cache Invalidation — The Hard Problem
  • TTL (Time-To-Live): Simplest — expire after N seconds. Risk: stale data during TTL window.
  • Event-driven: Invalidate on write events via Kafka/pub-sub. Complex but consistent.
  • Cache Stampede: Many requests hit DB simultaneously on cache miss. Fix: probabilistic early expiry or mutex/lock.

Circuit Breaker Pattern

public class Base62Encoder {

        "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789";

    public static String encode(long id) {
        StringBuilder sb = new StringBuilder();
        while (id > 0) {
            sb.append(CHARS.charAt((int)(id % 62)));
            id /= 62;
        }
        return sb.reverse().toString();
    }

Chunking Strategy — Key to Efficient Sync
// Files are split into 4MB chunks before upload
// Each chunk identified by SHA-256 hash (content-addressable storage)

public class FileChunker {
    private static final int CHUNK_SIZE = 4 * 1024 * 1024; // 4 MB

    public List<Chunk> chunk(File file) throws Exception {
        List<Chunk> chunks = new ArrayList<>();
        try (FileInputStream fis = new FileInputStream(file)) {
            byte[] buffer = new byte[CHUNK_SIZE];
            int bytesRead;
            int index = 0;
            while ((bytesRead = fis.read(buffer)) != -1) {
                byte[] data = Arrays.copyOf(buffer, bytesRead);
                String hash = sha256(data);
                chunks.add(new Chunk(index++, hash, data));
            }
        }
        return chunks;
    }
}
// Upload: check which chunks already exist (dedup) → only upload new chunks
// Delta sync: compare chunk hashes → only re-upload changed chunks

// Bloom Filter for URL deduplication
public class URLBloomFilter {
    private final BitSet   bits;
    private final int      size;
    private static final int NUM_HASH_FUNCTIONS = 7;

    public URLBloomFilter(int expectedInsertions) {
        // Optimal size: -n*ln(p) / (ln2)^2 where p=0.01 (1% false positive)
        this.size = expectedInsertions * 10;  // ~10 bits per element
        bits = new BitSet(size);
    }

    public void add(String url) {
        for (int i = 0; i < NUM_HASH_FUNCTIONS; i++)
            bits.set(hash(url, i) % size);
    }

    public boolean mightContain(String url) {
        for (int i = 0; i < NUM_HASH_FUNCTIONS; i++)
            if (!bits.get(hash(url, i) % size)) return false;
        return true;   // might be present (false positive possible)
    }
}
// 1B URLs × 10 bits = 10 billion bits = 1.25 GB — fits in memory!
