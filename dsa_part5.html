MediumGrid DFS

1. Number of Islands

▼
🧠 Logic Building
Grid as graph: Each '1' cell is a node. Edges connect adjacent '1' cells (up/down/left/right). An island = a connected component.
Strategy: Scan every cell. When you find a '1', increment island count and DFS to mark all connected '1's as visited (flip to '0'). Next unvisited '1' = new island.
Mark visited in-place by overwriting '1'→'0' — no separate visited array needed. This is the standard interview trick.
Pattern unlock: "Count connected components in grid" → DFS from each unvisited node, sink the island
Grid
11110
11010
11000
00000
Output
1
// DRY RUN: grid[0][0]='1' → island++, DFS sinks all connected 1s
// After DFS: entire top-left mass → '0'. No more '1's → return 1 ✓

public int numIslands(char[][] grid) {
    int count = 0;
    for (int r=0; r<grid.length; r++)
        for (int c=0; c<grid[0].length; c++)
            if (grid[r][c] == '1') { count++; dfs(grid,r,c); }
    return count;
}
private void dfs(char[][] g, int r, int c) {
    if (r<0||r>=g.length||c<0||c>=g[0].length||g[r][c]!= '1') return;
    g[r][c] = '0'; // sink: mark visited in-place
    dfs(g,r+1,c); dfs(g,r-1,c); dfs(g,r,c+1); dfs(g,r,c-1);
}
Time: O(R×C)Space: O(R×C) stack
MediumDFS + HashMap

2. Clone Graph

▼
🧠 Logic Building
Deep copy a graph: Every node needs a new clone, and all neighbor references must point to the NEW clones, not the originals.
HashMap as visited + mapping: Map<original, clone>. Before recursing into a neighbor, check if it already has a clone in the map — this prevents infinite loops on cycles.
DFS recipe: Create clone → store in map → recurse into each neighbor → add neighbor clones to current clone's list.
Pattern unlock: "Graph deep copy with cycle handling" → HashMap to track original→clone, DFS
Graph (adj list)
1:[2,4], 2:[1,3], 3:[2,4], 4:[1,3]
Output
Deep copy with same structure
// DRY RUN: clone(1)
//   map={1→clone1}. For neighbor 2: not in map → clone(2)
//     map={1→c1,2→c2}. For neighbor 1: IN map → use c1. For neighbor 3: clone(3)...
//   c2.neighbors=[c1,c3]. c1.neighbors=[c2,c4] etc. ✓

Map<Node,Node> map = new HashMap<>();

public Node cloneGraph(Node node) {
    if (node == null) return null;
    if (map.containsKey(node)) return map.get(node); // already cloned
    Node clone = new Node(node.val);
    map.put(node, clone);                              // store BEFORE recursing (cycle guard)
    for (Node nb : node.neighbors)
        clone.neighbors.add(cloneGraph(nb));           // recurse, get cloned neighbor
    return clone;
}
Time: O(V+E)Space: O(V)
MediumTopo Sort / Cycle Detection

3. Course Schedule — Can Finish?

▼
🧠 Logic Building
Model as directed graph: [a,b] means "to take a, first take b" → edge b→a. "Can finish all courses?" = "Is the graph a DAG (no cycle)?"
Kahn's Algorithm (BFS topo sort): Compute in-degree for every node. Add all nodes with in-degree 0 to queue (no prerequisites). Process queue: for each node, reduce neighbors' in-degree by 1; if neighbor's in-degree hits 0, add to queue.
Cycle detection: If we processed all N nodes → no cycle → can finish. If queue empties before processing all N → cycle exists → can't finish.
Pattern unlock: "Detect cycle in directed graph / dependency ordering" → Kahn's BFS topo sort, count processed nodes
n=2, prereqs
[[1,0]] (take 0 before 1)
Output
true (0 then 1)
// DRY RUN: n=2, prereqs=[[1,0]] → graph: 0→1
// inDegree=[0,1]. Queue=[0] (in-degree 0)
// Process 0: neighbor 1 → inDegree[1]=0 → queue=[1]. count=1
// Process 1: no neighbors. count=2
// count==n → no cycle → return true ✓
//
// Cycle case: [[1,0],[0,1]] → 0→1, 1→0
// inDegree=[1,1]. Queue=[] (no zero in-degrees) → stuck → count=0 ≠ 2 → false ✓

public boolean canFinish(int n, int[][] prereqs) {
    List<List<Integer>> adj = new ArrayList<>();
    int[] inDeg = new int[n];
    for (int i=0;i<n;i++) adj.add(new ArrayList<>());
    for (int[] p : prereqs) { adj.get(p[1]).add(p[0]); inDeg[p[0]]++; }
    Queue<Integer> q = new LinkedList<>();
    for (int i=0;i<n;i++) if(inDeg[i]==0) q.offer(i);
    int count = 0;
    while (!q.isEmpty()) {
        int node = q.poll(); count++;
        for (int nb : adj.get(node))
            if (--inDeg[nb] == 0) q.offer(nb);
    }
    return count == n; // processed all = no cycle
}
Time: O(V+E)Space: O(V+E)

4. Course Schedule II — Return Order

▼
🧠 Logic Building
Same Kahn's BFS as Course Schedule I — just collect the processed nodes into a result array instead of counting them.
Each node we dequeue is "safe to take" (all prerequisites done). The order we dequeue them = a valid course ordering.
Pattern unlock: "Topological ordering of DAG" → Kahn's BFS, collect dequeue order
n=4, prereqs
[[1,0],[2,0],[3,1],[3,2]]
Output
[0,1,2,3] or [0,2,1,3]
// DRY RUN: graph: 0→[1,2], 1→[3], 2→[3]
// inDeg=[0,1,1,2]. Queue=[0]
// Process 0 → res=[0]. neighbors 1,2: inDeg=[0,0,0,2] → queue=[1,2]
// Process 1 → res=[0,1]. neighbor 3: inDeg[3]=1 → queue=[2]
// Process 2 → res=[0,1,2]. neighbor 3: inDeg[3]=0 → queue=[3]
// Process 3 → res=[0,1,2,3] ✓

public int[] findOrder(int n, int[][] prereqs) {
    List<List<Integer>> adj = new ArrayList<>();
    int[] inDeg = new int[n];
    for (int i=0;i<n;i++) adj.add(new ArrayList<>());
    for (int[] p : prereqs) { adj.get(p[1]).add(p[0]); inDeg[p[0]]++; }
    Queue<Integer> q = new LinkedList<>();
    for (int i=0;i<n;i++) if(inDeg[i]==0) q.offer(i);
    int[] res = new int[n]; int idx=0;
    while(!q.isEmpty()) {
        int node=q.poll(); res[idx++]=node;
        for(int nb:adj.get(node)) if(--inDeg[nb]==0) q.offer(nb);
    }
    return idx==n ? res : new int[0]; // empty = cycle
}
Time: O(V+E)Space: O(V+E)

5. Network Delay Time — Dijkstra's Algorithm

▼
🧠 Logic Building
Single-source shortest path on weighted directed graph. Find the minimum time for a signal to reach all nodes from source k. Answer = max of all shortest distances.
Dijkstra template: dist[] initialized to ∞, dist[source]=0. Use a min-heap of (distance, node). Greedily pick the node with smallest known distance, relax its neighbors.
Relaxation: If dist[node] + edge_weight < dist[neighbor] → update dist[neighbor] and push to heap.
Why min-heap? We always want to process the currently closest unvisited node — greedy choice that guarantees optimality when all weights are non-negative.
Pattern unlock: "Shortest path in weighted graph" → Dijkstra: min-heap + dist[] relaxation
times, n=4, k=2
[[2,1,1],[2,3,1],[3,4,1]]
Output
2
// DRY RUN: source=2, graph: 2→1(1), 2→3(1), 3→4(1)
// dist=[∞,∞,0,∞,∞] (1-indexed). heap=[(0,2)]
// Pop (0,2): relax 1→dist[1]=1, relax 3→dist[3]=1. heap=[(1,1),(1,3)]
// Pop (1,1): no outgoing edges. heap=[(1,3)]
// Pop (1,3): relax 4→dist[4]=2. heap=[(2,4)]
// Pop (2,4): no edges.
// dist=[∞,1,0,1,2]. max(1,0,1,2)=2 ✓

public int networkDelayTime(int[][] times, int n, int k) {
    Map<Integer,List<int[]>> adj = new HashMap<>();
    for (int[] t : times) adj.computeIfAbsent(t[0], x->new ArrayList<>()).add(new int[]{t[1],t[2]});
    int[] dist = new int[n+1];
    Arrays.fill(dist, Integer.MAX_VALUE);
    dist[k] = 0;
    PriorityQueue<int[]> pq = new PriorityQueue<>((a,b)->a[0]-b[0]); // min-heap by dist
    pq.offer(new int[]{0,k});
    while(!pq.isEmpty()) {
        int[] cur=pq.poll();
        int d=cur[0], u=cur[1];
        if(d>dist[u]) continue; // stale entry
        for(int[] nb : adj.getOrDefault(u, Collections.emptyList())) {
            int nd=d+nb[1];
            if(nd<dist[nb[0]]) { dist[nb[0]]=nd; pq.offer(new int[]{nd,nb[0]}); }
        }
    }
    int ans=0;
    for(int i=1;i<=n;i++) { if(dist[i]==Integer.MAX_VALUE) return -1; ans=Math.max(ans,dist[i]); }
    return ans;
}
Time: O((V+E) log V)Space: O(V+E)
⚠ Dijkstra fails with negative weights. Use Bellman-Ford for those.

6. Pacific Atlantic Water Flow

▼
🧠 Logic Building
Naive approach: For each cell, DFS to check if water can reach both oceans. O(R²C²) — too slow.
Reverse thinking: Instead of water flowing DOWN from cell to ocean, think of water flowing UP from ocean to cells. BFS/DFS from all Pacific border cells, then from all Atlantic border cells. Any cell reachable from BOTH = answer.
Reverse condition: In reverse flow, water can go from cell A to cell B if B.height >= A.height (uphill = reverse of downhill).
Pattern unlock: "Reverse the flow direction → multi-source BFS from boundaries"
Heights grid (5×5)
[[1,2,2,3,5],[3,2,3,4,4],...]
Output
Cells reachable from both oceans
// BFS from Pacific (top row + left col) → pac[][] reachable
// BFS from Atlantic (bottom row + right col) → atl[][] reachable
// Answer: cells where pac[r][c] && atl[r][c]

public List<List<Integer>> pacificAtlantic(int[][] h) {
    int R=h.length, C=h[0].length;
    boolean[][] pac=new boolean[R][C], atl=new boolean[R][C];
    Queue<int[]> pq=new LinkedList<>(), aq=new LinkedList<>();
    for(int r=0;r<R;r++){
        pq.offer(new int[]{r,0});  pac[r][0]=true;   // left col → Pacific
        aq.offer(new int[]{r,C-1}); atl[r][C-1]=true; // right col → Atlantic
    }
    for(int c=0;c<C;c++){
        pq.offer(new int[]{0,c});  pac[0][c]=true;   // top row → Pacific
        aq.offer(new int[]{R-1,c}); atl[R-1][c]=true; // bottom row → Atlantic
    }
    bfs(h,pq,pac,R,C); bfs(h,aq,atl,R,C);
    List<List<Integer>> res=new ArrayList<>();
    for(int r=0;r<R;r++) for(int c=0;c<C;c++)
        if(pac[r][c]&&atl[r][c]) res.add(Arrays.asList(r,c));
    return res;
}
private void bfs(int[][] h, Queue<int[]> q, boolean[][] vis, int R, int C) {
    int[][] dirs={{0,1},{0,-1},{1,0},{-1,0}};
    while(!q.isEmpty()) {
        int[] cur=q.poll(); int r=cur[0],c=cur[1];
        for(int[] d:dirs) {
            int nr=r+d[0],nc=c+d[1];
            if(nr>=0&&nr<R&&nc>=0&&nc<C&&!vis[nr][nc]&&h[nr][nc]>=h[r][c]) {
                vis[nr][nc]=true; q.offer(new int[]{nr,nc});
            }
        }
    }
}
Time: O(R×C)Space: O(R×C)
Union-Find (Disjoint Set Union)
Union-Find answers "are these two nodes connected?" in near O(1) time. The magic: path compression + union by rank.

🌟 The Universal DSU Template

int[] parent, rank;
void init(int n) {
    parent=new int[n]; rank=new int[n];
    for(int i=0;i<n;i++) parent[i]=i; // each node is its own parent
}
int find(int x) {
    if(parent[x]!=x) parent[x]=find(parent[x]); // path compression
    return parent[x];
}
boolean union(int x, int y) {
    int px=find(x), py=find(y);
    if(px==py) return false; // already connected → adding this edge = cycle
    if(rank[px]<rank[py]) { int t=px;px=py;py=t; }
    parent[py]=px;                         // union by rank
    if(rank[px]==rank[py]) rank[px]++;
    return true;
}
MediumDSU Components

7. Number of Connected Components in Undirected Graph

▼
🧠 Logic Building
Start with n separate components (every node is its own island).
For each edge [u,v]: If u and v are in different components → union them → components-- . If already connected → skip (would create cycle).
Final answer: The remaining number of components.
Pattern unlock: "Dynamic connectivity → components count" → DSU: start=n, decrement on each successful union
n=5, edges
[[0,1],[1,2],[3,4]]
Output
2
// DRY RUN: n=5, components=5
// union(0,1): different → merge → components=4
// union(1,2): different → merge → components=3
// union(3,4): different → merge → components=2
// return 2 ✓

public int countComponents(int n, int[][] edges) {
    int[] parent=new int[n], rank=new int[n];
    for(int i=0;i<n;i++) parent[i]=i;
    int comps=n;
    for(int[] e:edges) if(union(parent,rank,e[0],e[1])) comps--;
    return comps;
}
// (use find/union helpers from template above)
Time: O(N·α(N))Space: O(N)
MediumDSU Cycle Edge

8. Redundant Connection — Find the Cycle Edge

▼
🧠 Logic Building
Given a tree with one extra edge that creates a cycle — find that edge.
DSU insight: Process edges one by one. The first edge [u,v] where find(u)==find(v) means u and v are ALREADY connected — adding this edge creates the cycle. That edge is the answer.
If multiple such edges exist, return the last one in the input (problem constraint). Since we return immediately on first cycle, and the problem guarantees exactly one redundant edge, this always works.
Pattern unlock: "Find cycle-creating edge" → DSU: return first edge where find(u)==find(v)
edges
[[1,2],[1,3],[2,3]]
Output
[2,3]
// DRY RUN:
// [1,2]: find(1)=1, find(2)=2 → different → union. parent=[_,1,1,3]
// [1,3]: find(1)=1, find(3)=3 → different → union. parent=[_,1,1,1]
// [2,3]: find(2)=find(1)=1, find(3)=find(1)=1 → SAME → cycle! return [2,3] ✓

public int[] findRedundantConnection(int[][] edges) {
    int n=edges.length;
    int[] par=new int[n+1], rank=new int[n+1];
    for(int i=0;i<=n;i++) par[i]=i;
    for(int[] e:edges) {
        int pu=find(par,e[0]), pv=find(par,e[1]);
        if(pu==pv) return e;  // already connected → this edge is redundant
        if(rank[pu]<rank[pv]) { int t=pu;pu=pv;pv=t; }
        par[pv]=pu; if(rank[pu]==rank[pv]) rank[pu]++;
    }
    return new int[0];
}
int find(int[] p, int x){ return p[x]==x?x:(p[x]=find(p,p[x])); }
Time: O(N·α(N))Space: O(N)

9. Kth Largest Element in an Array

▼
🧠 Logic Building
Why min-heap of size K for "K largest"? Keep the K largest seen so far in a min-heap. The smallest of them (heap.peek()) is the Kth largest. When a new element arrives: if it's larger than the smallest in heap → it deserves a spot → push it, pop the smallest.
After processing all elements: heap.peek() = Kth largest overall.
Alternative — QuickSelect: O(N) average, O(N²) worst. Heap is O(N log K) and more predictable for interviews.
Pattern unlock: "Kth largest → min-heap of size K → peek is answer"
nums, k=2
[3,2,1,5,6,4]
Output
5
// DRY RUN: k=2, nums=[3,2,1,5,6,4]
// Add 3: heap=[3]
// Add 2: heap=[2,3]  (size=k=2)
// Add 1: 1 < peek(2) → skip
// Add 5: 5 > peek(2) → pop 2, push 5 → heap=[3,5]
// Add 6: 6 > peek(3) → pop 3, push 6 → heap=[5,6]
// Add 4: 4 < peek(5) → skip
// peek()=5 = 2nd largest ✓

public int findKthLargest(int[] nums, int k) {
    PriorityQueue<Integer> minH = new PriorityQueue<>(); // min-heap
    for (int n : nums) {
        minH.offer(n);
        if (minH.size() > k) minH.poll(); // evict smallest → keep K largest
    }
    return minH.peek(); // smallest of K largest = Kth largest
}
Time: O(N log K)Space: O(K)
MediumFreq Map + Min-Heap

10. Top K Frequent Elements

▼
🧠 Logic Building
Two steps: (1) Count frequencies with a HashMap. (2) Use a min-heap of size K ordered by frequency to find the K most frequent elements.
Same min-heap-of-size-K trick: Compare by frequency. If new element's frequency > smallest frequency in heap → replace it.
Bucket sort alternative: O(N) — create frequency buckets (index = freq, value = list of nums). Scan from high to low frequency. Great follow-up to mention.
Pattern unlock: "Top K by frequency" → HashMap + min-heap by frequency
nums, k=2
[1,1,1,2,2,3]
Output
[1,2]
// freq: {1:3, 2:2, 3:1}
// min-heap by freq, size k=2:
//   push [1,3]: heap=[[1,3]]
//   push [2,2]: heap=[[2,2],[1,3]]
//   push [3,1]: freq(3)=1 < peek freq=2 → skip
// Result from heap: [1,2] ✓

public int[] topKFrequent(int[] nums, int k) {
    Map<Integer,Integer> freq = new HashMap<>();
    for (int n:nums) freq.merge(n,1,Integer::sum);
    PriorityQueue<Integer> minH = new PriorityQueue<>((a,b)->freq.get(a)-freq.get(b));
    for (int n:freq.keySet()) {
        minH.offer(n);
        if(minH.size()>k) minH.poll();
    }
    return minH.stream().mapToInt(Integer::intValue).toArray();
}
Time: O(N log K)Space: O(N)
HardMin-Heap K Pointers

11. Merge K Sorted Linked Lists

▼
🧠 Logic Building
Brute force: Collect all values, sort, rebuild. O(N log N) where N = total nodes.
Better — min-heap on K list heads: Push the head of each list into a min-heap. Repeatedly: poll the minimum node → add to result → push that node's next (if non-null) into heap.
Why better? O(N log K) — the heap always has at most K elements. Much better than O(N log N) when K << N.
Pattern unlock: "Merge K sorted sequences" → Min-heap of K current pointers, poll+push-next
Lists
1→4→5, 1→3→4, 2→6
Output
1→1→2→3→4→4→5→6
// DRY RUN: heap initially=[1(L1),1(L2),2(L3)] (min-heap by val)
// Poll 1(L1) → result: 1. Push L1.next=4. heap=[1(L2),2(L3),4(L1)]
// Poll 1(L2) → result: 1→1. Push L2.next=3. heap=[2(L3),3(L2),4(L1)]
// Poll 2(L3) → result: 1→1→2. Push 6. heap=[3(L2),4(L1),6(L3)]
// ... continues until heap empty → 1→1→2→3→4→4→5→6 ✓

public ListNode mergeKLists(ListNode[] lists) {
    PriorityQueue<ListNode> pq = new PriorityQueue<>((a,b)->a.val-b.val);
    for (ListNode h:lists) if(h!=null) pq.offer(h); // init with all heads
    ListNode dummy=new ListNode(0), cur=dummy;
    while(!pq.isEmpty()) {
        cur.next=pq.poll();    // take minimum
        cur=cur.next;
        if(cur.next!=null) pq.offer(cur.next); // push next node of same list
    }
    return dummy.next;
}
Time: O(N log K)Space: O(K)
HardTwo Heaps

12. Find Median from Data Stream

▼
🧠 Logic Building
Two heaps partition the data at the median: A max-heap holds the lower half, a min-heap holds the upper half. The median is always at the tops of these heaps.
Invariant: (1) All elements in max-heap ≤ all elements in min-heap. (2) Sizes differ by at most 1. Always add to max-heap first, then rebalance.
Add(num): Push to max-heap. Pop max-heap's top → push to min-heap (ensures cross-heap ordering). If min-heap is larger → pop its top back to max-heap.
Pattern unlock: "Running median" → Max-heap (lower half) + min-heap (upper half), maintain balance
Stream
addNum(1), addNum(2), findMedian(), addNum(3), findMedian()
Output
1.5, 2.0
// DRY RUN:
// add(1): maxH=[1], minH=[]. rebalance: maxH=[1],minH=[]
// add(2): push 1 to maxH → maxH=[1,2]? No — push 2 to maxH, pop max(2) to minH
//         maxH=[1], minH=[2]. sizes equal → median=(1+2)/2=1.5 ✓
// add(3): push 3 to maxH → maxH=[1,3]? push 3, pop max(3) to minH=[2,3]
//         minH.size > maxH.size → pop min(2) back to maxH=[1,2]
//         maxH=[1,2], minH=[3]. maxH.size > → median=maxH.peek()=2.0 ✓

PriorityQueue<Integer> lo = new PriorityQueue<>(Collections.reverseOrder()); // max-heap
PriorityQueue<Integer> hi = new PriorityQueue<>();                          // min-heap

public void addNum(int num) {
    lo.offer(num);
    hi.offer(lo.poll());         // ensure max of lo ≤ min of hi
    if(hi.size()>lo.size()) lo.offer(hi.poll()); // balance sizes
}
public double findMedian() {
    return lo.size()>hi.size() ? lo.peek() : (lo.peek()+hi.peek())/2.0;
}
Add: O(log N)Find: O(1)

13. Meeting Rooms II — Minimum Rooms Required

▼
🧠 Logic Building
How many rooms needed = max overlapping meetings at any point.
Sort by start time. Use a min-heap of end times (end times of meetings currently in rooms). When a new meeting starts: if the earliest ending meeting is done (heap.peek() ≤ new start) → reuse that room (pop old, push new end). Otherwise → need a new room (just push new end).
Heap size = number of rooms in use at any time. Max heap size ever reached = answer.
Pattern unlock: "Minimum resources for overlapping intervals" → Sort by start, min-heap of end times
intervals
[[0,30],[5,10],[15,20]]
Output
2
// DRY RUN: sorted by start: [0,30],[5,10],[15,20]
// [0,30]:  heap empty → new room. heap=[30]
// [5,10]:  5 < peek(30) → overlap → new room. heap=[10,30]
// [15,20]: 15 > peek(10) → reuse! pop 10, push 20. heap=[20,30]
// Max heap size = 2 ✓

public int minMeetingRooms(int[][] intervals) {
    Arrays.sort(intervals, (a,b)->a[0]-b[0]); // sort by start
    PriorityQueue<Integer> endTimes = new PriorityQueue<>(); // min-heap of end times
    for(int[] iv:intervals) {
        if(!endTimes.isEmpty() && endTimes.peek()<=iv[0])
            endTimes.poll(); // room freed up — reuse it
        endTimes.offer(iv[1]);
    }
    return endTimes.size(); // rooms still in use = rooms needed
}
Time: O(N log N)Space: O(N)
MediumSort + Sweep

14. Merge Intervals

▼
🧠 Logic Building
Sort by start time. After sorting, any overlapping intervals must be adjacent. Walk through: if current interval overlaps with the last merged one (cur.start ≤ last.end) → extend last.end = max(last.end, cur.end). Otherwise → start a new merged interval.
Why sort first? Without sorting, you'd need to compare each interval against all others. Sorting reduces it to a single pass.
Pattern unlock: "Merge overlapping intervals" → Sort by start, linear sweep extending or starting
intervals
[[1,3],[2,6],[8,10],[15,18]]
Output
[[1,6],[8,10],[15,18]]
// DRY RUN: sorted: [1,3],[2,6],[8,10],[15,18]
// res=[[1,3]]
// [2,6]: 2≤3 → overlap → extend: res=[[1,6]]
// [8,10]: 8>6 → new: res=[[1,6],[8,10]]
// [15,18]: 15>10 → new: res=[[1,6],[8,10],[15,18]] ✓

public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals,(a,b)->a[0]-b[0]);
    List<int[]> res = new ArrayList<>();
    for(int[] iv:intervals) {
        if(res.isEmpty() || res.get(res.size()-1)[1] < iv[0])
            res.add(iv);                       // no overlap → new interval
        else
            res.get(res.size()-1)[1] = Math.max(res.get(res.size()-1)[1],iv[1]); // extend
    }
    return res.toArray(new int[0][]);
}
Time: O(N log N)Space: O(N)
MediumGreedy Reach

15. Jump Game — Can Reach End?

▼
🧠 Logic Building
Greedy insight: Track the farthest index reachable so far. For each position i (up to current max reach), update maxReach = max(maxReach, i + nums[i]).
If at any point i > maxReach → we can't reach position i → stuck → return false.
If maxReach ≥ last index → return true. No need to simulate actual jumps.
Pattern unlock: "Reachability in array" → Greedy: track farthest reachable index
nums
[2,3,1,1,4]
Output
true
// DRY RUN: [2,3,1,1,4]
// i=0: maxReach=max(0,0+2)=2
// i=1: 1≤2 ✓ maxReach=max(2,1+3)=4
// i=2: 2≤4 ✓ maxReach=max(4,2+1)=4
// i=3: 3≤4 ✓ maxReach=max(4,3+1)=4
// i=4: 4≤4 ✓ maxReach=max(4,4+4)=8 ≥ last index(4) → true ✓
//
// DRY RUN fail: [3,2,1,0,4]
// i=3: maxReach=max(3,3+0)=3. i=4: 4>3 → false ✓

public boolean canJump(int[] nums) {
    int maxReach = 0;
    for (int i=0; i<nums.length; i++) {
        if (i > maxReach) return false;              // can't reach here
        maxReach = Math.max(maxReach, i + nums[i]); // extend reach
    }
    return true;
}
Time: O(N)Space: O(1)

16. Non-overlapping Intervals — Min Removals

▼
🧠 Logic Building
Equivalent to: Find maximum number of non-overlapping intervals → answer = total - max non-overlapping. This is the classic "Activity Selection" greedy problem.
Greedy rule: Sort by end time. Always greedily pick the interval that ends earliest (leaves most room for future intervals). If next interval overlaps with current → remove it (count removal). If no overlap → update current end to new interval's end.
Why sort by end? Picking the earliest-ending interval maximizes the remaining time window for subsequent intervals.
Pattern unlock: "Max non-overlapping = Activity Selection" → Sort by end, greedily keep earliest-ending non-overlapping
intervals
[[1,2],[2,3],[3,4],[1,3]]
Output
1 (remove [1,3])
// DRY RUN: sort by end: [1,2],[2,3],[1,3],[3,4]
// end=2, removals=0
// [2,3]: start(2)≥end(2) → no overlap → end=3
// [1,3]: start(1)

public int eraseOverlapIntervals(int[][] intervals) {
    Arrays.sort(intervals,(a,b)->a[1]-b[1]); // sort by END time
    int removals=0, end=Integer.MIN_VALUE;
    for(int[] iv:intervals) {
        if(iv[0] < end) removals++;           // overlap → remove this interval
        else           end = iv[1];            // keep it, update end
    }
    return removals;
}
Time: O(N log N)Space: O(1)
💡 The greedy choice of "sort by end time" appears in Meeting Rooms, Activity Selection, and Non-overlapping Intervals. Recognise it as one family.
